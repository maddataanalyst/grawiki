# Plan: Split the example notebook into focused tutorials and make notebook tooling optional

## Goal
Create three focused tutorial notebooks from `notebooks/example_notebook.ipynb`, keep them aligned with the current `GraphRAG` public API, and move Jupyter / notebook visualization dependencies into explicit optional installs in `pyproject.toml`.

## Findings
- `notebooks/example_notebook.ipynb` currently mixes four concerns in one flow: setup, ingestion, deduplication, and agent memory.
- The current `GraphRAG` facade already exposes notebook-friendly step methods in `src/grawiki/rag/graph_rag.py`:
  - `read_document`
  - `chunk_document`
  - `embed_document`
  - `embed_chunks`
  - `build_document_node`
  - `build_chunk_nodes`
  - `persist_document_and_chunks`
  - `extract_kg_per_chunk`
  - `persist_entities_and_relationships`
  - plus `search`, `dedupe_entities`, `remember`, and `recall`
- The notebook sample texts already exist under `notebooks/experimental_data/*.txt`.
- `pyproject.toml` currently has only one package extra, `docs`, while notebook-related tooling is partly hidden in dependency groups (`research`) and some imports used by the notebook are not declared there (`python-dotenv`, `rich`, `tqdm`).
- Visualization libraries are not currently declared.

## Implementation plan

### 1. Replace the monolithic notebook with three tutorial notebooks
Add three numbered notebooks under `notebooks/` with a simple progression and explicit prerequisites.

Planned notebook set:
1. `notebooks/01_ingest_and_deduplicate.ipynb`
2. `notebooks/02_agent_memory_and_recall.ipynb`
3. `notebooks/03_visualize_graph.ipynb`

The original `example_notebook.ipynb` should either be removed after verification or reduced to a short pointer cell to the new notebook set. The safer implementation path is to keep it until the new notebooks are validated, then decide whether to remove it in the same change.

### 2. Notebook 1: single-document ingestion, inspection, duplicate finding, deduplication, and querying
Build notebook 1 around one Medium article text file and the public step-by-step API.

Notebook structure:
1. Intro markdown:
   - explain that the sample texts are Medium.com articles by Filip Wójcik
   - include `https://medium.com/@filip.igor.wojcik`
   - mention they are fully available without subscription
   - explain required extras and FalkorDB dependency
2. Environment and imports:
   - `autoroot`
   - `dotenv.load_dotenv`
   - `rich.print`
   - `Path`
   - `FalkorGraphDB`
   - `GraphRAG`
   - optionally `tqdm` only where it adds value
3. Database initialization:
   - use a notebook-local DB path such as `notebooks/local_falcor.db`
   - initialize `GraphRAG` with `resolve_entities_on_ingest=True`
4. Stepwise ingestion using one source document:
   - `read_document`
   - `chunk_document`
   - `embed_document`
   - `embed_chunks`
   - `build_document_node`
   - `build_chunk_nodes`
   - `persist_document_and_chunks`
   - `extract_kg_per_chunk`
   - `persist_entities_and_relationships`
5. Database inspection:
   - show a few persisted documents / chunks / entities using FalkorDB read queries
   - show `list_entities()` output
   - optionally show `neighbor_relationships()` for one entity
6. Existing-entity query examples:
   - use `rag.search(...)`
   - include one direct database query for entities or relationships for transparency
7. Duplicate detection:
   - run `find_entity_duplicate_candidates(...)` or `dedupe_entities(..., dry_run=True)` first
   - display candidate groups and merge reports clearly
8. Deduplication execution:
   - run `dedupe_entities(..., dry_run=False)`
   - re-query entities to show the graph after merge
9. End-state querying:
   - demonstrate the higher-level facade with `rag.search(...)`
   - optionally also demonstrate one lower-level helper-backed database query for comparison
10. Cleanup cell:
   - remind users to call `database.close()`

Content choices:
- Use a single article for the ingestion walkthrough to keep runtime predictable.
- Use the remaining article texts as optional extension cells or as notes for notebook 2 / 3.
- Prefer string IDs everywhere when linking nodes; avoid the old notebook’s incorrect `list[int]` memory-link pattern.

### 3. Notebook 2: PydanticAI agent with retrieval tool and memory tool on top of an existing graph DB
Assume notebook 1 has already built the FalkorDB graph and that the DB is reused as-is.

Notebook structure:
1. Intro markdown:
   - state the prerequisite: run notebook 1 first
   - restate the Medium article source note
2. Setup:
   - open the existing FalkorDB path
   - construct `GraphRAG`
   - construct a `pydantic_ai.Agent`
3. Tooling:
   - `search_knowledge_graph(query: str, limit: int = 5)` calling `await rag.search(...)`
   - `remember_user_fact(memory_text: str, user_id: str, related_node_ids: list[str] | None = None)` calling `await rag.remember(..., metadata={"user_id": user_id}, related_node_ids=...)`
   - `recall_user_memories(query: str, user_id: str, hops: int = 1, limit: int = 5)` calling `await rag.recall(...)`
4. Result serialization:
   - convert `NodeHit` and nested `Node` objects with Pydantic model serialization, not `dataclasses.asdict`
   - ensure outputs shown in notebook are readable and stable
5. Demo conversation flow:
   - ask a factual question about the ingested graph
   - ask a user-profile question where the agent learns something about the user
   - persist the learned fact with the memory tool
   - ask a follow-up question that uses `recall` to recover the user information
6. Direct recall inspection:
   - explicitly call `rag.recall(...)` in a separate cell
   - show `node.properties["recall_context"]`
7. Optional DB inspection:
   - show stored `__memory__` nodes and their graph links via FalkorDB query
8. Cleanup:
   - `database.close()`

Important fixes relative to the old notebook:
- Add a real recall tool instead of only claiming memory recall in the system prompt.
- Stop using `dataclasses.asdict` on `NodeHit`.
- Use `Sequence[str]`-compatible node IDs for memory links.
- Prefer `remember` with raw text for the main demo path unless a `MemoryNode` is being intentionally taught.

### 4. Notebook 3: visualize the constructed graph
Make notebook 3 read from the already-populated FalkorDB graph and build a lightweight visual view of the graph.

Recommended approach:
- Use `networkx` to build an in-memory graph from queried nodes and relationships.
- Use `matplotlib` for a static visualization because it is lightweight and easy to keep optional.

Notebook structure:
1. Intro markdown:
   - prerequisite: notebook 1 has already built the graph
2. Setup and imports:
   - `networkx`
   - `matplotlib.pyplot`
   - existing DB / `GraphRAG` imports if useful
3. Query graph data:
   - fetch a manageable subset of nodes and relationships from FalkorDB
   - likely exclude most chunk/document nodes in the first visualization to reduce clutter
4. Build a NetworkX graph:
   - add nodes with labels, names, and node-type metadata
   - add edges with relationship labels
5. Plot a readable graph:
   - color by node type (`__entity__`, `__memory__`, `__chunk__`, `__document__` or filtered entity-only view)
   - draw relationship labels only when graph size is small enough
6. Optional focused views:
   - one ego graph around a chosen entity
   - one subgraph around a memory node to illustrate recall context visually
7. Cleanup:
   - `database.close()`

Design constraints:
- Keep the default visualization to a small, readable subset.
- Prefer a deterministic node selection strategy so repeated notebook runs look similar.
- If querying the full graph is noisy, the notebook should build an entity-centric or memory-centric subgraph instead of a full dump.

### 5. Refactor `pyproject.toml` so notebook tooling is truly optional
Move notebook-specific dependencies into package extras instead of relying on the `research` dependency group alone.

Recommended extras:
- `notebooks`
  - `autoroot`
  - `ipykernel`
  - `ipywidgets`
  - `python-dotenv`
  - `rich`
  - `tqdm`
- `viz`
  - `networkx`
  - `matplotlib`

Notes:
- Keep `docs` as-is.
- Keep `dev` as a dependency group for contributor tooling.
- Keep `falkordb` as-is unless there is a broader packaging change requested later.
- Remove or repurpose the `research` group so notebook installation guidance does not point to a half-complete dependency set.
- Installation guidance should likely become:
  - base development: `uv sync --group dev`
  - FalkorDB support: `uv sync --group falkordb`
  - notebooks: `uv sync --extra notebooks`
  - visualization: `uv sync --extra viz`
  - combined notebook experience: `uv sync --group falkordb --extra notebooks --extra viz`

## Validation plan
1. Validate notebook JSON structure after writing the `.ipynb` files.
2. Run targeted lint / formatting checks if any notebook-adjacent Python files are changed.
3. Run `uv run pytest` to ensure code/package changes did not break the existing suite.
4. If environment variables for the configured models are available, perform a smoke run of the notebook flows or execute representative cells manually.
5. Verify that a notebook-oriented install works with optional dependencies, ideally with a fresh sync command such as:
   - `uv sync --group falkordb --extra notebooks --extra viz`

## Risks and watchpoints
- Notebook execution depends on model credentials and external services, so fully automated execution may not be possible in CI.
- Deduplication results can vary slightly depending on extraction and embeddings, so notebook output should present the process, not promise exact duplicate counts.
- Full-graph visualization may become unreadable quickly; entity-focused or ego-graph views are likely the right default.
- The old notebook used patterns that no longer match the current API, so the rewrite should avoid carrying over stale code blindly.

## Implementation order
1. Refactor `pyproject.toml` extras and installation guidance.
2. Create notebook 1 around the public step API and deduplication workflow.
3. Create notebook 2 around search, remember, and recall.
4. Create notebook 3 around optional visualization.
5. Update README/docs/CODEMAP as needed.
6. Run validation and then decide whether to keep or retire `example_notebook.ipynb`.
