# MkDocs And Read the Docs Plan

## Goal

Set up a public documentation site for GraWiki using MkDocs and Read the Docs, with content that:

- introduces the project as early work in progress,
- explains GraWiki as an attempt to connect knowledge graphs with Andrej Karpathy's LLM Wiki idea,
- includes Mermaid diagrams for core flows such as document ingestion,
- supports bibliography-backed citations from a BibTeX file,
- enables the `mkdocs-copy-to-llm` plugin,
- keeps docs tooling optional so it is not required for normal package installation,
- makes `docs/` the official public documentation source,
- moves the current internal-only markdown material out of `docs/` into a new `agent_tools/` directory.

## Current Repository Findings

- Packaging is managed from `pyproject.toml` and currently uses `uv` dependency groups: `dev`, `falkordb`, and `research`.
- There is no existing `mkdocs.yml` or `.readthedocs.yaml`.
- `README.md` is short and can seed the public overview, but it needs cleanup and expansion.
- The existing top-level `docs/` directory is currently an internal-maintenance area, not a clean public docs source.
  - It contains `docs/CODEMAP.md` and several planning or strategy markdown files.
- You now want that internal material relocated into `agent_tools/` so that `docs/` becomes the official public documentation tree.
- `src/grawiki/rag/graph_rag.py` exposes a clear ingestion pipeline that is suitable for diagramming:
  - `read_document`
  - `chunk_document`
  - `embed_document`
  - `embed_chunks`
  - `build_document_node`
  - `build_chunk_nodes`
  - `persist_document_and_chunks`
  - `extract_kg_per_chunk`
  - `persist_entities_and_relationships`
  - high-level `ingest` and `ingest_text`

## Recommended Implementation Choices

### 1. Public docs source directory

Use `docs/` as the MkDocs source directory for the official public site.

Reason:

- it matches your stated intent that `docs/` should become the official documentation,
- it avoids introducing an unnecessary second public-docs directory such as `site_docs/`,
- it gives a conventional layout for MkDocs and Read the Docs.

### 2. Internal documentation relocation

Create a new top-level `agent_tools/` directory and move all current files from `docs/` into it before creating the public docs pages.

Reason:

- separates public documentation from internal agent or contributor artifacts,
- prevents internal planning material from being published,
- leaves `docs/` dedicated to the official documentation site.

Recommended migration approach:

- move every current markdown file from `docs/` into `agent_tools/`,
- preserve filenames unless a collision or naming cleanup is necessary,
- update any internal references that still point at `docs/CODEMAP.md` or other moved files,
- treat `agent_tools/` as internal-only repository guidance and planning content.

### 3. Docs dependency model

Use a formal package extra:

- add `[project.optional-dependencies]`,
- define `docs = [...]`,
- install docs tooling with `uv sync --extra docs` or `pip install .[docs]`.

Reason:

- this matches the future packaging direction where GraWiki can expose installable subsets such as `grawiki[rag]`,
- docs tooling remains optional and is not part of the base install,
- the dependency story becomes consistent across docs and future feature extras,
- Read the Docs can install from the docs extra.

Recommended shape:

- base `project.dependencies` stays focused on the default package experience,
- optional extras are introduced for non-core concerns and future modular surfaces,
- docs is the first extra, implemented in a way that will not block later additions like `rag`, `falkordb`, or similar feature-focused extras.

Note:

- the repo can still keep `dependency-groups.dev` for contributor tooling such as `pytest`, `ruff`, and `pre-commit`,
- docs should not be duplicated in a `docs` dependency group unless there is a concrete workflow need to maintain both forms.

### 4. MkDocs stack

Use a minimal and stable stack:

- `mkdocs`
- `mkdocs-material`
- `mkdocs-mermaid2-plugin`
- `mkdocs-copy-to-llm`
- `mkdocs-bibtex`
- `pymdown-extensions`

Planned MkDocs configuration should include:

- Material theme,
- `docs_dir: docs`,
- plugins: `search`, `mermaid2`, `copy-to-llm`, `bibtex`,
- Mermaid fence support through `pymdownx.superfences` custom fences,
- navigation for overview, concepts, architecture or flows, and references.

### 5. Read the Docs integration

Add:

- `mkdocs.yml`
- `.readthedocs.yaml`

Recommended RTD configuration:

- config version 2,
- Python 3.12,
- MkDocs build target using `mkdocs.yml`,
- install docs dependencies through `uv` using the `docs` extra.

## Content Plan

### Tone and framing

The docs should describe GraWiki as:

- an early-stage open source Python project,
- a work in progress rather than a finished framework,
- an experiment in combining knowledge-graph extraction and retrieval with agent memory or wiki-style knowledge for LLMs.

### Core overview points

The overview should communicate:

- GraWiki extracts structured knowledge from documents into a graph,
- the graph supports retrieval and context expansion,
- the project also stores agent memories as graph nodes,
- the long-term framing is knowledge organization for LLM systems rather than only classic RAG.

### Diagram plan

Primary Mermaid diagram:

- source file
- `read_document`
- `chunk_document`
- `embed_document` and `embed_chunks`
- `build_document_node` and `build_chunk_nodes`
- `persist_document_and_chunks`
- `extract_kg_per_chunk`
- optional `resolve_entities_on_ingest`
- `persist_entities_and_relationships`
- resulting graph state

Potential secondary diagram:

- query input
- retriever strategies
- graph search primitives
- neighbor expansion or memory recall
- response context assembly

### Citation plan

Use the supplied BibTeX entries to support statements about:

- knowledge graphs and graph construction,
- GraphRAG and KG-enhanced RAG,
- AI engineering and GenAI application framing,
- knowledge graphs plus LLMs.

## Risks And Mitigations

### Risk: internal markdown gets published accidentally

Mitigation:

- move all current `docs/` files into `agent_tools/` before adding public pages,
- keep `mkdocs.yml` pointed only at `docs/`,
- avoid symlinks or mixed public and internal content in the same tree.

### Risk: future extras become inconsistent

Mitigation:

- start with `[project.optional-dependencies]` now so docs and later feature extras follow one packaging model.

### Risk: moved internal references break

Mitigation:

- audit repository references to `docs/` paths during implementation,
- update `AGENTS.md` and any internal markdown references to `agent_tools/` equivalents.

### Risk: bibliography plugin syntax differences

Mitigation:

- keep citations simple and validate with a local strict MkDocs build.

### Risk: Mermaid rendering issues with theme integration

Mitigation:

- configure `pymdownx.superfences` exactly for Mermaid custom fences and verify in local build output.

## Finalized Packaging Decision

Docs should be implemented as a formal optional dependency extra under `[project.optional-dependencies].docs`.

This keeps documentation tooling out of the default install and establishes the packaging pattern needed for future modular installs such as `grawiki[rag]`.
