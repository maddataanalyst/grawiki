# Plan: Add API Reference Docs with mkdocstrings

## Goal

Extend the public MkDocs site with an API reference section powered by `mkdocstrings[python]` and `mkdocs-autorefs`, with good discoverability for:

- the `GraphRAG` facade,
- retrieval primitives,
- graph/domain models,
- database abstractions,
- entity similarity inspection,
- duplicate-candidate discovery,
- deduplication helpers.

The new docs should work with the current MkDocs + Read the Docs setup and remain compatible with the Cinder theme.

## Current State

- `mkdocs.yml` is minimal and currently includes only hand-written pages:
  - `index.md`
  - `concepts.md`
  - `flows.md`
  - `references.md`
- The docs extra in `pyproject.toml` does not yet include `mkdocstrings` or `mkdocs-autorefs`.
- The codebase already has a meaningful public/documentable surface in:
  - `grawiki.__init__` / `GraphRAG`
  - `grawiki.retrieval.*`
  - `grawiki.graph.models`
  - `grawiki.db.base`
  - `grawiki.similarity.*`
- Similarity and deduplication are already first-class features in code, especially through:
  - `GraphRAG.find_entity_duplicate_candidates(...)`
  - `GraphRAG.dedupe_entities(...)`
  - `GraphRAG` ingest-time resolution options
  - `EntitySimilarityFinder`
  - matcher implementations and result models
  - `grawiki.similarity.deduplication`

## Implementation Steps

### 1. Add docs dependencies

Update `pyproject.toml` docs extra to include:

- `mkdocstrings[python]`
- `mkdocs-autorefs`

Keep existing docs tooling (`mkdocs`, `mkdocs-cinder`, `mkdocs-bibtex`, `mkdocs-copy-to-llm`, `mkdocs-mermaid2-plugin`, `pymdown-extensions`).

Expected follow-up:

- refresh `uv.lock`
- verify the RTD build still works through the existing `docs` extra installation path

### 2. Configure MkDocs for API references

Update `mkdocs.yml` to:

- enable `autorefs`
- enable `mkdocstrings`
- configure Python handler search paths so modules resolve from `src/`
- keep existing bibtex, mermaid, and Cinder theme setup intact

Recommended mkdocstrings behavior:

- show public members by default
- hide private members
- hide implementation internals where possible
- prefer signatures and docstrings, not source dumps
- use inventory/autorefs support so symbols can be linked from narrative docs

### 3. Add new docs pages under `docs/api/`

Create hand-written overview pages plus lightweight `mkdocstrings` wrapper pages.

Recommended content strategy:

- `docs/api/index.md`
  - explain how to read the API docs
  - point users first to `GraphRAG`
  - explain which sections are stable vs advanced
- `docs/api/graphrag.md`
  - hand-written intro to the facade
  - link major flows: ingest, search, remember, recall
  - include duplicate inspection and deduplication methods explicitly
  - embed `::: grawiki.rag.graph_rag.GraphRAG`
- `docs/api/similarity/index.md`
  - high-level explanation of the two-stage duplicate workflow:
    - semantic-key collisions
    - broader similarity matching
    - optional merge/deduplication
  - prominently link:
    - `GraphRAG.find_entity_duplicate_candidates`
    - `GraphRAG.dedupe_entities`
    - `EntitySimilarityFinder`
    - result model pages
- remaining pages mostly wrap one module/class each with a short intro

### 4. Make similarity/deduplication discoverable

This is the main user-requested emphasis area and should not be left as raw API only.

The implementation should ensure the docs surface all of the following clearly:

- `EntitySimilarityFinder`
  - exact semantic-key collision detection
  - broader similarity scanning
  - combined duplicate-candidate reporting
- matchers
  - `VectorEntitySimilarityMatcher`
  - `RapidFuzzEntitySimilarityMatcher`
  - `EntitySimilarityMatcher` protocol/base
- result models
  - `EntityDuplicateCandidates`
  - `EntitySimilarityResult`
  - `SemanticKeyCollisionCandidates`
- deduplication helpers
  - `MergeReport`
  - `pick_master`
  - `build_merged_master`
- facade-level convenience methods on `GraphRAG`
  - duplicate inspection
  - dedupe/merge execution
  - ingest-time entity resolution

### 5. Clean up docstrings that will render poorly in API docs

Before exposing modules prominently via `mkdocstrings`, fix stale or confusing docstrings where the generated pages would otherwise be misleading.

Highest-value cleanup targets:

- `src/grawiki/core/embedding.py`
  - module docstring appears stale relative to the current embedding approach
- `src/grawiki/graph/extraction.py`
  - parameter wording reportedly references `embedder` while constructor uses `embedding`
- `src/grawiki/retrieval/text.py`
  - docstring reportedly uses `embedder` wording instead of `embedding`
- `src/grawiki/rag/graph_rag.py`
  - review `search(...)` docs for stale wording around default retrieval behavior and limits
- `src/grawiki/db/base.py`
  - review any stale singular `label` wording vs current `labels`
- package `__init__.py` files that are currently empty/minimal:
  - `src/grawiki/rag/__init__.py`
  - `src/grawiki/retrieval/__init__.py`
  - `src/grawiki/graph/__init__.py`

The goal is not to document every internal helper, but to ensure the rendered public reference is coherent and accurate.

### 6. Keep advanced/internal boundaries clear

Avoid exposing too much low-level plumbing as if it were the primary public API.

Specific cautions:

- `grawiki.db.falkordb` should be presented as an advanced adapter page, not a first-stop public page
- private helpers and underscore-prefixed functions should be hidden from generated docs
- `grawiki.graph.extraction` should be labeled advanced because some extracted/intermediate structures are workflow internals
- `grawiki.db.cypher` and prompt modules should stay out of public API nav

### 7. Validate the docs build end-to-end

After implementation:

- install/sync docs dependencies with the updated docs extra
- run `uv run mkdocs build --strict`
- verify API pages render under Cinder
- verify `autorefs` links resolve across hand-written pages and generated API pages
- verify Read the Docs config still works with the extra and new plugins

## Notes for Implementation

- Prefer a hybrid documentation style:
  - hand-written overview pages for narrative and discoverability
  - `mkdocstrings` for the symbol reference itself
- Lead users toward `GraphRAG` first, not low-level modules
- Make similarity/deduplication a dedicated section so the feature is easy to find
- Keep generated docs selective; avoid turning every internal module into public documentation
- Because Cinder is now the active theme, implementation should verify code blocks, heading depth, and API tables still render acceptably without Material-specific assumptions

## Acceptance Criteria

- MkDocs includes a visible `API` section
- API pages render using `mkdocstrings` on local strict build
- Public references for similarity finders and deduplication are easy to discover
- `GraphRAG` API page includes duplicate-candidate and dedupe-related methods
- Cross-links between narrative docs and API symbols resolve via `mkdocs-autorefs`
- Read the Docs remains buildable through the existing docs extra
